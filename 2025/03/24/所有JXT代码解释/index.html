JXT

发表于2025-03-24|更新于2025-03-24
|总字数:3.3k|阅读时长:14分钟|浏览量:

Setup_JXT类

下面用一个小例子来说明：

假设 CSV 文件内容（表头在第 0 行）

ID Name Age Dept Location
ID Name Age Dept Location
1 Alice 30 HR NYC
2 Bob 25 IT SF
3 Carol 30 IT NYC
  • key_column = 2 （“Name”、“Age” 是关键属性）
  • join_column = 2 （“Dept”、“Location” 是 join 属性）
  • record_num = 3

第一步：构建 reverse_id（属性–值 → 记录行号）

属性–值 对应行号
NameAlice [1]
NameBob [2]
NameCarol [3]
Age30 [1,3]
Age25 [2]
DeptHR [1]
DeptIT [2,3]
LocationNYC [1,3]
LocationSF [2]

第二步：构造 XSet（||表示拼接）

对于每条记录 & 每个 join 列（Dept、Location）：

  1. 计算 xInd = SHA256(K_x ∥ ID ∥ joinColumnName)
  2. 计算 yInd = SHA256(K_y ∥ joinColumnValue)
  3. 将 xInd ⊕ yInd → long → 插入 Bloom filter

结果：Bloom filter 中包含 6 个值（3 行 × 2 join 列）。

第三步：构造 TSet

对 reverse_id 中每个“属性–值” w：

  1. 计算 token = SHA256(K_token ∥ w ∥ table_id)
  2. 对列表中的每个行号 i：
    • 计算 h_x 和 h_y（使用 SHA256(K_h1/h2 ∥ w ∥ cnt) 与对应 xInd/yInd 异或）
    • 加密该行 ID（AES_encrypt(SHA256(K_aes ∥ w), ID)）
    • 将它们打包成一个 tuple
  3. 把 token → List 放到 tset

最终效果

  • XSet（Bloom filter）可以快速判断某个 join 值是否存在于表中
  • TSet 存储了每个属性–值 对应哪些行，并对行 ID 做了加密和混淆

这样，在查询时，只需提供一个属性–值 对（比如 Age30），客户端就能：

  1. 计算对应 token → 在 TSet 找到所有 tuple

  2. 解密 tuple 得到原始行 ID

  3. 使用 XSet 验证 join 关联
    无需暴露表中所有原始数据！

Sever_JXT类

例：

1️⃣ 已知数据（服务器端）

表 token tuple 索引 h_x/h_y 值 ct（加密 RecordID）
A TokenA 0 Y_A1 “A1”
A TokenA 1 Y_A2 “A2”
B TokenB 0 X_B1 “B1”
B TokenB 1 X_B2 “B2”

• Table A 的 tuple 存在于 tset.get(TokenA)

• Table B 的 tuple 存在于 tset.get(TokenB)

Bloom filter f_2（Table B 的 XSet）记录了两个 long 值：

  • long(Y_A1 ⊕ X_B1)
  • long(Y_A2 ⊕ X_B2)

原因：

服务器只检查由 Table A 和 Table B 的 tuple 组合生成的 XOR 结果，是否存在于 Table B 的 Bloom filter 中。

为什么只用 Table B 的 XSet？

  • 设计决定：JXT 查询流程中，服务器只需要用第二张表（Table B）的 Bloom filter 来判断两个 tuple 是否在 join 属性上相同。
  • 对称性：如果 Table A 的 tuple 和 Table B 的 tuple 在 join 属性上匹配，那么它们的 XOR 值（Y_A ⊕ X_B）一定同时存在于 Table A 的 XSet 和 Table B 的 XSet。但只检查一个就足够避免假阳性。

示例回顾

  • Table A XSet（f_1）也包含 long(Y_A1⊕X_B1) 和 long(Y_A2⊕X_B2)
  • Table B XSet（f_2）也包含相同两个值

但在 search() 里，代码只调用了

因此只用了 Table B 的 Bloom filter 来做最终过滤判定，节省一次检查而不影响正确性。

2️⃣ 客户端提供的输入

参数 内容
stag1 TokenA (指 Table A)
stag2 TokenB (指 Table B)
token1 [t1_1, t1_2] 解混淆 Table A h_y
token2 [t2_1, t2_2] 解混淆 Table B h_x
join_column 0 （只有一个 join 属性）

对于 Table A 的每条 tuple i：

  • j=0 → xtoken2[0] = X_B1 ⊕ t2_1
  • j=1 → xtoken2[1] = X_B2 ⊕ t2_2

JXT主程序

例子：

场景：学生表 (Table1) & 课程表 (Table2) 通过 CourseID 做 Join

Table1 (Students)
StudentID Name CourseID
S001 Tom C101
S002 Mary C102
Table2 (Courses)
CourseID Title Credits
C101 Math 3
C102 Bio 4

运行结果示例（首次输出）

  • S001,C101：表示学生 Tom（ID=S001）选修课程 C101
  • S002,C102：表示学生 Mary（ID=S002）选修课程 C102
  • res size : 4：返回了两对匹配，每对包含两个加密 ID，所以总长度是 4
  • average search time：1000 次查询的平均耗时，大约 0.045 毫秒

这个例子中，keyword1 = "CourseIDC101" 和 keyword2 = "CourseIDC101" 表示我们在两个表中都查询 CourseID=C101；同理 CourseID=C102 会在后续循环中自动测试（因为代码运行 1000 次）。
